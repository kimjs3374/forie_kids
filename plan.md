# 프로젝트 명세서 및 실행 계획: 아파트 어린이 놀이터 월별 예약 관리 시스템

## 0. 기존 문서 대비 주요 보완점
기존 문서는 "예약 접수 폼" 수준으로는 충분했지만, 실제 운영 기준인 **월별 예약 시스템**을 구현하기에는 아래 내용이 부족했다.

- 월별 예약의 핵심 개념인 **예약 대상 월**(`target_month`) 정의가 없었음
- 실제 예약 가능 단위인 **날짜/시간 슬롯** 또는 **회차** 개념이 빠져 있었음
- 정원, 중복 예약 방지, 접수 기간, 마감 정책 등 **운영 룰**이 정의되지 않았음
- 관리자 기능이 단순 조회 수준이라 **월 오픈/마감, 슬롯 관리, 상태 변경**이 불가능했음
- 실제 제작을 위한 **개발 단계, 우선순위, 산출물, 일정**이 부족했음

따라서 본 문서는 **"월별 예약 관리가 가능한 실제 제작용 MVP 명세서"** 기준으로 재정리한다.

---

## 1. 프로젝트 개요
아파트 입주민이 매월 어린이 놀이터 이용 가능 일정을 확인하고 예약할 수 있도록 하는 웹 애플리케이션을 개발한다.

시스템은 크게 두 영역으로 나뉜다.

1. **입주민용 예약 페이지**
   - 현재 예약 가능한 월을 확인
   - 해당 월의 날짜/시간 슬롯을 선택
   - 예약 정보를 입력하고 접수

2. **관리자용 백오피스**
   - 공지사항 관리
   - 예약 대상 월 생성/오픈/마감
   - 월별 슬롯 생성 및 정원 관리
   - 예약 내역 조회/상태 관리
   - 엑셀 다운로드

핵심은 단순 신청서가 아니라, **월 단위 운영 정책을 반영한 예약 시스템**이라는 점이다.

---

## 2. 프로젝트 목표

### 2.1. 비즈니스 목표
- 입주민이 관리사무소 문의 없이 월별 예약 가능 현황을 직접 확인할 수 있게 한다.
- 관리자 입장에서 예약 접수, 마감, 정원 체크, 월별 명단 다운로드를 간소화한다.
- 공지/운영 변경 사항을 즉시 반영할 수 있는 운영형 서비스로 만든다.

### 2.2. MVP 목표
- 한 달 단위 예약 운영이 가능해야 한다.
- 관리자가 월별 예약 일정을 직접 열고 닫을 수 있어야 한다.
- 정원이 초과되지 않도록 예약 가능 여부를 제어해야 한다.
- 월별 예약 리스트를 검색/필터/엑셀 다운로드할 수 있어야 한다.

## 3. MVP 범위 정의

### 3.1. 입주민 기능
- 상단 공지사항 확인
- 현재 예약 가능한 월 목록 확인
- 특정 월의 예약 가능 슬롯(날짜/시간/정원 잔여) 확인
- 예약 정보 입력 후 예약 신청
- 예약 완료/실패 메시지 확인

### 3.2. 관리자 기능
- 관리자 로그인/로그아웃
- 공지사항 등록 및 수정
- 예약 대상 월 생성, 수정, 오픈, 마감
- 월별 슬롯 생성/수정/비활성화
- 예약 목록 조회(월별, 날짜별, 상태별)
- 예약 상태 변경(예: 예약완료, 취소)
- 엑셀 다운로드

### 3.3. MVP 제외 항목(2차 확장)
- 문자/카카오 알림 발송
- 입주민 직접 예약 조회/취소 마이페이지
- 다국어 지원
- 관리자 권한 레벨 분리(슈퍼관리자/운영자)
- 결제 기능

## 4. 핵심 운영 정책 (MVP 기준)
실제 제작 가능성을 위해 아래 정책을 기본값으로 정의한다.

| 항목 | MVP 기본 정책 | 비고 |
|---|---|---|
| 예약 단위 | 월별로 오픈된 슬롯 예약 | 예: 2026-04 월 예약 |
| 예약 대상 | 예약 오픈 상태인 월만 신청 가능 | `OPEN` 상태 |
| 슬롯 단위 | 날짜 + 시작시간 + 종료시간 | 예: 2026-04-12 / 10:00~11:00 |
| 정원 관리 | 슬롯별 정원(capacity) 관리 | 잔여석 0이면 예약 불가 |
| 중복 예약 | 동일 슬롯 중복 예약 금지 | 휴대폰번호/동호수 기준 검증 |
| 동일 월 예약 횟수 | 기본 1회(권장), 설정으로 완화 가능 | 운영 정책에 따라 변경 가능 |
| 예약 확정 방식 | 접수 즉시 예약 완료 | 추후 승인제로 확장 가능 |
| 취소 처리 | 입주민 직접 취소 미지원, 관리자만 처리 | MVP 단순화 목적 |
| 접수 기간 | 월별 오픈일시 ~ 마감일시 사이만 허용 | 관리자 설정 |

> 위 정책은 구현 난이도와 운영 편의성을 고려한 **실행 가능한 기본안**이다. 세부 정책이 미정이어도 이 기준으로 즉시 제작을 시작할 수 있다.

## 5. 사용자 시나리오

### 5.1. 입주민 예약 시나리오
1. 입주민이 메인 페이지에 접속한다.
2. 상단에서 공지사항과 현재 예약 가능한 월을 확인한다.
3. 원하는 월을 선택한다.
4. 해당 월의 예약 가능한 날짜/시간 슬롯 목록을 확인한다.
5. 슬롯을 선택한 뒤 이름, 동/호수, 연락처, 아이 인원수, 개인정보 동의를 입력한다.
6. 시스템은 중복 예약, 마감 여부, 정원 초과 여부를 검사한다.
7. 유효하면 예약을 저장하고 완료 메시지를 보여준다.

### 5.2. 관리자 운영 시나리오
1. 관리자는 `/admin`에서 로그인한다.
2. 다음 달 예약을 위해 예약 대상 월을 생성한다.
3. 해당 월의 날짜/시간 슬롯과 정원을 등록한다.
4. 예약 시작 시점에 월 상태를 `OPEN`으로 변경한다.
5. 접수 현황을 확인하면서 필요 시 공지를 수정하거나 예약 상태를 조정한다.
6. 월 마감 후 엑셀로 예약자 명단을 다운로드한다.

## 7. 기술 스택 및 구현 방식

### 7.1. 백엔드
- **Language:** Python 3.12
- **Framework:** Flask
- **ORM:** SQLAlchemy
- **Migration:** Flask-Migrate 또는 Alembic
- **Authentication:** Flask-Login

### 7.2. 데이터베이스
- **DB:** Supabase PostgreSQL
- 월/슬롯/예약 구조를 정규화하여 운영

### 7.3. 프론트엔드
- **Template Engine:** Jinja2
- **UI:** Bootstrap 5 기반 반응형 화면
- 복잡한 SPA보다는 서버 렌더링 방식으로 MVP를 빠르게 완성

### 7.4. 부가 라이브러리
- `python-dotenv`: 환경변수 관리
- `psycopg2-binary`: PostgreSQL 연결
- `openpyxl` 또는 `pandas`: 엑셀 다운로드
- `Flask-WTF`: 폼 및 CSRF 보호 권장

## 8. 권장 URL 구조

### 8.1. 입주민 페이지
- `GET /` : 메인 예약 페이지
- `POST /reserve` : 예약 접수
- `GET /months/<yyyy-mm>` : 특정 월 슬롯 조회(선택 구현)

### 8.2. 관리자 페이지
- `GET/POST /admin` : 로그인
- `GET /admin/dashboard` : 대시보드
- `POST /admin/notice` : 공지사항 수정
- `GET/POST /admin/months` : 예약 월 등록/조회
- `POST /admin/months/<id>/status` : 월 상태 변경
- `GET/POST /admin/slots` : 슬롯 관리
- `GET /admin/reservations` : 예약 목록
- `POST /admin/reservations/<id>/status` : 예약 상태 변경
- `GET /admin/reservations/export` : 엑셀 다운로드
- `GET /admin/logout` : 로그아웃

## 9. 데이터베이스 스키마 설계

### 9.1. `admins`
관리자 로그인 계정 테이블

| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | Serial PK | 관리자 ID |
| username | Varchar Unique | 로그인 아이디 |
| password_hash | Varchar | 해시된 비밀번호 |
| created_at | Timestamp | 생성일시 |
| last_login_at | Timestamp NULL | 마지막 로그인 |

### 9.2. `settings`
공지사항 및 시스템 단일 설정값 저장

| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | Integer PK | 기본값 1 |
| notice_text | Text | 상단 공지사항 |
| policy_json | JSONB NULL | 운영 정책 확장용 |
| updated_at | Timestamp | 수정일시 |

### 9.3. `reservation_months`
예약 운영 단위가 되는 월 정보

| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | Serial PK | 월 ID |
| target_month | Varchar(7) Unique | 예: `2026-04` |
| title | Varchar | 운영 제목(선택) |
| open_at | Timestamp | 예약 오픈 일시 |
| close_at | Timestamp | 예약 마감 일시 |
| status | Varchar | `DRAFT` / `OPEN` / `CLOSED` |
| max_reservations_per_household | Integer NULL | 세대당 월 최대 예약 횟수 |
| created_at | Timestamp | 생성일시 |
| updated_at | Timestamp | 수정일시 |

### 9.4. `reservation_slots`
월에 속한 실제 예약 가능 슬롯 정보

| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | Serial PK | 슬롯 ID |
| month_id | Integer FK | `reservation_months.id` |
| play_date | Date | 이용 날짜 |
| start_time | Time | 시작 시간 |
| end_time | Time | 종료 시간 |
| capacity | Integer | 총 정원 |
| status | Varchar | `ACTIVE` / `INACTIVE` / `CLOSED` |
| created_at | Timestamp | 생성일시 |
| updated_at | Timestamp | 수정일시 |

### 9.5. `reservations`
실제 예약 신청 데이터

| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | Serial PK | 예약 ID |
| month_id | Integer FK | 예약 대상 월 |
| slot_id | Integer FK | 선택한 슬롯 |
| name | Varchar | 예약자 이름 |
| apt_unit | Varchar | 동/호수 |
| phone | Varchar | 연락처 |
| children_count | Integer | 아이 인원수 |
| consent_agreed | Boolean | 개인정보 동의 여부 |
| status | Varchar | `RESERVED` / `CANCELLED` |
| created_at | Timestamp | 신청일시 |
| updated_at | Timestamp | 수정일시 |

### 9.6. 인덱스 및 검증 권장 사항
- `reservation_months.target_month` 유니크
- `reservation_slots(month_id, play_date)` 인덱스
- `reservations(month_id, slot_id)` 인덱스
- 동일 슬롯 중복 예약 방지를 위한 앱 레벨 검증
- 동일 월 1회 제한 정책이 활성화되면 `apt_unit` 또는 `phone` 기준 추가 검증

## 10. 핵심 비즈니스 로직

### 10.1. 예약 가능 조건
아래 조건을 모두 만족해야 예약 가능하다.

1. 대상 월 상태가 `OPEN`이어야 함
2. 현재 시간이 `open_at ~ close_at` 범위 내여야 함
3. 슬롯 상태가 `ACTIVE`여야 함
4. 슬롯 정원이 남아 있어야 함
5. 필수 입력값이 모두 유효해야 함
6. 개인정보 동의가 체크되어야 함
7. 동일 슬롯 중복 예약이 아니어야 함
8. 월별 최대 예약 횟수 정책을 초과하지 않아야 함

### 10.2. 예약 완료 처리
- 예약 저장
- 성공 메시지 표시
- 관리자 화면에 즉시 반영
- 엑셀 다운로드 대상에 포함

### 10.3. 관리자 상태 변경
- 월 상태: `DRAFT → OPEN → CLOSED`
- 예약 상태: `RESERVED ↔ CANCELLED`
- 슬롯 상태: `ACTIVE / INACTIVE / CLOSED`

## 11. 보안 및 운영 가이드

### 11.1. 환경 변수
민감 정보는 코드에 하드코딩하지 않는다.

```env
FLASK_SECRET_KEY=your_secret_key_here
DATABASE_URL=postgresql://postgres:********@db.xxxxx.supabase.co:5432/postgres
ADMIN_USERNAME=admin
ADMIN_PASSWORD=change_me
```

### 11.2. 보안 권장사항
- 관리자 비밀번호는 평문 저장 금지, 반드시 해시 저장
- 관리자 페이지는 로그인 세션 필수
- CSRF 방어 적용
- 서버 사이드 입력 검증 적용
- 연락처/개인정보는 로그에 그대로 남기지 않기
- 엑셀 다운로드는 관리자만 가능하도록 제한

### 11.3. 운영 권장사항
- 월 오픈 전에 테스트 슬롯으로 사전 검증
- 마감 후 데이터 백업 또는 엑셀 보관

## 16. 최종 결론
이 프로젝트는 단순 신청 폼이 아니라, **관리자가 월 단위로 예약을 열고 닫으며 슬롯과 정원을 관리하는 월별 예약 시스템**으로 정의되어야 한다.

따라서 실제 구현은 아래 순서를 따르면 된다.

1. 월(`reservation_months`) 구조를 먼저 만든다.
2. 월에 속한 슬롯(`reservation_slots`)을 만든다.
3. 입주민은 열린 월의 활성 슬롯만 예약한다.
4. 관리자는 공지, 월 상태, 슬롯, 예약 내역을 백오피스에서 관리한다.
5. 운영 결과는 엑셀로 다운로드한다.

이 문서 기준으로 바로 개발에 들어가면, **정책이 완전히 세부 확정되지 않았더라도 MVP는 충분히 제작 가능**하다.
