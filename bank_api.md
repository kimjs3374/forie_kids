# 프로젝트 요약: BankAPI 연동 자동 입금 확인 및 전광판 시스템 구축

## 1. 목표
BankAPI를 활용하여 무통장 입금 내역을 주기적으로 조회하고, 자체 DB의 주문 데이터와 대조하여 '자동 결제 완료' 처리를 수행한다. 미확인 입금건은 관리자 승인을 거쳐 메인 화면 전광판에 노출한다.

## 2. 최고 수준 보안 아키텍처 (DB 암호화 저장 방식)
금융 정보(계좌 비밀번호, 생년월일/사업자번호 등)를 `.env` 파일에 직접 평문으로 하드코딩하는 것을 **엄격히 금지**한다. 대신 다음 구조를 구현한다.

* **.env 역할 축소:** `.env` 파일에는 데이터를 암호화/복호화하기 위한 `MASTER_SECRET_KEY`(강력한 랜덤 문자열)와 `BANK_API_KEY` 같은 서비스 연동 키만 저장한다.
* **DB 암호화 저장:** 은행 계좌번호, 생년월일, 계좌 비밀번호 등은 관리자 페이지 UI를 통해 입력받는다. 입력받은 민감 정보는 `MASTER_SECRET_KEY`를 이용해 **양방향 암호화(AES-256 등)** 처리한 뒤 데이터베이스(시스템 설정 테이블)에 저장한다.
* **인메모리 복호화:** 스케줄러가 BankAPI를 호출할 때만 DB에서 암호화된 값을 가져와 메모리 상에서 일시적으로 복호화하여 사용하고, 호출이 끝나면 즉시 메모리에서 해제한다.
* **로그 마스킹:** API 통신 과정의 로깅 시 민감 정보(계좌번호, 비밀번호 등)가 로그 파일에 남지 않도록 반드시 마스킹(`***`) 처리한다.

## 3. API 호출 및 스케줄러(Cron) 정책
* **호출 한도 방어 및 스케줄링 (월 3,000건 / 31일 기준 최적화):**
  * **기준 시간:** KST (한국 표준시) 기준
  * **제외 시간대 (API 호출 중지):** 02:00 ~ 07:00 (총 5시간)
  * **호출 주기:** 가동 시간(19시간) 동안 **12분 주기**로 실행. (월 최대 약 2,945회 호출로 한도 준수)
* **호출 최적화:** 무조건 전체 내역을 가져오지 않고, 자체 DB에 저장된 **가장 최근 거래일시 이후의 데이터만** API로 요청하여 DB에 적재한다.

## 4. 데이터베이스 스키마 요구사항
기존 주문 테이블(`Order`) 외에 다음 구조를 추가/생성한다.

**[Table: bank_settings (계좌 연동 설정 테이블)]**
* BankAPI '계좌 거래내역 조회'에 필요한 필수 파라미터들을 저장하는 테이블.
* (예: 은행코드, 계좌번호, 암호화된 비밀번호, 암호화된 생년월일/사업자번호 등)

**[Table: bank_transactions (거래 내역 적재 테이블)]**
* `id`: PK
* `deposit_name`: 입금자명
* `amount`: 입금 금액
* `transaction_date`: 은행 거래 일시
* `status`: 상태값 (`PENDING` / `MATCHED` / `UNMATCHED` / `IGNORED`)
* `matched_order_id`: 매칭 성공 시 주문 ID (FK, Nullable)
* `is_billboard_approved`: 전광판 노출 승인 여부 (Boolean, 기본값 `False`)
* `created_at`: DB 적재 일시 (12개월 경과 시 자동 삭제 로직 추가)

## 5. 핵심 매칭 엔진 로직 (Matching Engine)
DB에 새롭게 적재된 `status = PENDING` 상태의 거래내역을 대상으로 다음을 실행한다.

**[매칭 조건] (3가지 모두 충족 시)**
1. **금액 일치:** `bank_transactions.amount` == `Order.total_amount`
2. **이름 일치:** 양쪽 데이터의 공백을 모두 제거한 후 비교
3. **시간 유효성:** `bank_transactions.transaction_date` >= `Order.created_at`

**[매칭 결과 처리]**
* **성공 시:** `Order` 결제 완료 처리, `bank_transactions` 상태 `MATCHED` 업데이트 및 주문 ID 맵핑.
* **실패/예외 시:** `bank_transactions` 상태 `UNMATCHED` 업데이트 (관리자 수동 처리 대기).

## 6. 관리자(Admin) 페이지 요구사항
* **[신규] 은행 계좌 연동 설정 페이지:** BankAPI 공식 문서(https://bankapi.co.kr/docs)의 '계좌 거래내역 조회' API 명세서를 분석하여, 호출에 필수적인 데이터(은행코드, 계좌번호, 계좌 비밀번호, 주민/사업자번호 등)를 입력받을 수 있는 UI 폼을 구현한다. 저장 시 2번 항목의 보안 정책(양방향 암호화)을 반드시 적용해야 한다.
* **리스트 뷰:** 미확인 입금 내역(`UNMATCHED`) 목록 노출.
* **수동 매칭:** 특정 주문 검색 및 강제 매칭 기능.
* **전광판 승인:** `is_billboard_approved`를 `True`로 토글하는 버튼.
* **보류/무시:** 회사 공금 등은 `IGNORED` 처리.

## 7. 메인 화면 전광판 (Frontend & Backend 로직)
* **노출 조건:** `status == UNMATCHED` AND `is_billboard_approved == True`
* **마스킹 처리 (중요):** 백엔드에서 데이터 전송 전 입금자명 마스킹 (예: 홍길동 -> 홍*동, 남궁민수 -> 남**수).
* **출력 포맷:** `[월/일] [마스킹된 이름]님 [금액]원 입금자를 찾습니다.`
